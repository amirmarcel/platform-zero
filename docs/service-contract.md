# The Service Contract

Every service on this platform is declared in exactly one hand-written file:
`services/<name>/service.yaml`.

That file is the contract between the engineer and the platform. The engineer promises
to declare these things. The platform promises that, given them, the service will be
deployed, monitored, alerted on, and rollback-able without the engineer writing a
single line of Kubernetes YAML.

## The manifest

```yaml
# services/checkout-api/service.yaml
apiVersion: platform/v1
name: checkout-api
owner: payments-team          # a team, never a person
tier: 1                       # 1 = user-facing critical, 2 = internal, 3 = batch/async

image:
  repository: ghcr.io/example/checkout-api
  tag: v1.4.2                 # immutable tag; :latest is rejected

runtime:
  port: 8080
  replicas: 3
  resources:
    requests: { cpu: 100m, memory: 128Mi }
    limits:   { cpu: 500m, memory: 512Mi }
  probes:
    readiness: /healthz/ready
    liveness:  /healthz/live

slo:
  availability: 99.9          # percent, over the window
  latency_p99_ms: 300
  window: 30d

operations:
  runbook: docs/runbooks/checkout-api.md
  rollback: gitops-revert     # the only supported strategy in v1
```

Every field above is **required**. There are no optional fields in v1, and that is the
point: optional fields become unset fields, and unset fields become the 3am page with
no owner.

## The standard, and how each rule is enforced

A checklist that lives in a wiki is a preference. A checklist that fails a pull request
is a standard. Each rule below maps to the mechanism that actually enforces it.

| # | Every service must have… | Enforced by |
|---|---|---|
| 1 | An owning **team** (not an individual) | `platformctl validate` — schema + team must exist in `platform/teams.yaml` |
| 2 | A **tier** (1/2/3) | Schema enum; drives alert routing and SLO defaults |
| 3 | An **immutable image tag** | `platformctl validate` — rejects `latest` and unpinned tags |
| 4 | **Resource requests and limits** | Schema-required; validate rejects limits < requests |
| 5 | **Readiness and liveness probes** | Schema-required; rendered into the Deployment |
| 6 | An **SLO** with an explicit window | Schema-required; generates the recording + alert rules |
| 7 | A **dashboard** | Generated from the manifest — cannot be missing |
| 8 | A **runbook** that the alert links to | `platformctl validate` — fails if the referenced file does not exist |
| 9 | A **rollback strategy** | Schema enum; v1 supports `gitops-revert` only |
| 10 | **No drift** between manifest and deployed artifacts | CI regenerates all derived files and fails on diff |

Rule 10 is the one that makes the other nine hold. Without it, the standard erodes the
first time someone hand-edits a rendered manifest to ship a hotfix.

## What the platform derives from the manifest

One declaration in, four artifacts out. None of these are hand-written; all are
regenerated by `platformctl render` and diff-checked in CI.

| Derived artifact | Path | What the manifest drives |
|---|---|---|
| **Helm values** | `helm/service/values/<name>.yaml` | image, replicas, port, resources, probes |
| **ArgoCD Application** | `argocd/apps/<name>.yaml` | source path, destination namespace, sync policy |
| **Prometheus rules** | `observability/rules/<name>.yaml` | SLI recording rules + burn-rate alert from the SLO block |
| **Grafana dashboard** | `observability/dashboards/<name>.json` | latency, error rate, saturation panels + SLO/error-budget panel |

The alert generated in row 3 carries `runbook_url` and `owner` as annotations, taken
straight from the manifest. That is the mechanical link between rule 1, rule 8, and the
page that fires at 3am: the alert *cannot* exist without knowing who owns it and what
to do about it, because the manifest could not have validated without both.

## The golden path

What an engineer actually does to ship a new service:

```
platformctl init checkout-api      # scaffolds service.yaml with sane tier-based defaults
$EDITOR services/checkout-api/service.yaml
platformctl render checkout-api    # generates the four derived artifacts
git commit && open a PR            # CI validates schema + checks for drift
                                   # merge → ArgoCD syncs → dashboard and SLO alert exist
```

No Kubernetes manifests written. No dashboard requested. No monitoring ticket filed.

## Versioning

The manifest carries `apiVersion: platform/v1`. When the contract changes — a new
required field, a new rollback strategy — the version increments and `platformctl`
supports both versions during the migration window. Breaking the contract silently
across a fleet of services is how platform teams lose the trust of their users, and
trust is the only reason anyone stays on the golden path.
